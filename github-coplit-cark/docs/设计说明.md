# GitHub Copilot 人机等待桥接方案

## 1. 目标

这个项目要解决的问题是：

- 让 `GitHub Copilot` 在一次用户提问触发的工作流中，遇到缺失信息时不要直接结束回答
- 改为调用一个 `MCP` 工具，进入“等待人工输入”的状态
- 人工输入不依赖聊天窗口本身，而是通过一个独立的网页完成
- 网页提交后，LLM 可以继续调用工具拿到输入，再接着完成后续任务

第一目标平台是：

- `GitHub Copilot`

需要兼容的客户端是：

- `VS Code`
- `Codex CLI`
- `Claude Code CLI`
- `OpenCode`

## 2. 设计原则

### 2.1 不依赖客户端专有交互能力

不同客户端虽然都支持 `MCP`，但不一定都支持相同的交互 UI 能力，例如：

- 内置表单
- elicitation
- 挂起后恢复同一轮工具调用

所以这个方案不把“用户输入”绑定在某个客户端的聊天 UI 里，而是统一走外部网页。

### 2.2 提示词只负责约束行为，不负责实现等待

“等待用户输入”真正发生的位置不是提示词，而是：

- MCP 工具
- 外部网页
- 会话状态机

提示词的职责是告诉模型：

- 什么时候必须调用工具
- 什么时候不能直接收尾
- 什么时候应该继续轮询

### 2.3 会话状态必须由桥接层自己维护

不能把会话持久化寄托在客户端内部状态上，因为：

- `VS Code`
- `Codex CLI`
- `Claude Code CLI`
- `OpenCode`

它们的内部会话机制并不统一，也不对外暴露稳定的可复用存储接口。

所以桥接层要自己维护：

- `session_id`
- 当前状态
- 用户输入
- 上下文摘要
- 过期时间
- 客户端标识

## 3. 总体架构

系统分成 4 个部分：

### 3.1 提示词层

分两层：

- 全局系统提示词
- MCP 工具返回里的内置控制指令

作用是让模型在多轮工具调用里依然记得：

- 需要继续调用 `workflow_poll`
- 不能过早输出最终答案
- 只有状态满足条件时才能结束任务

### 3.2 MCP 服务层

MCP 服务对客户端暴露统一工具。

当前最小设计只暴露 2 个工具：

- `workflow_wait_for_user`
- `workflow_poll`

### 3.3 本地或远程 Web 输入页

当模型发现缺少人工输入时，MCP 工具返回一个 `ui_url`。

用户通过这个页面输入内容，而不是在聊天框中继续补充。

这个设计的好处是：

- 客户端无关
- 容易单独控制权限和样式
- 可以扩展成表单、审批页、确认页

### 3.4 会话状态机

状态机负责保存和推进整个等待过程。

当前核心状态包括：

- `waiting_user`
- `submitted`
- `expired`
- `failed`

后续如果接入完整任务执行器，还可以扩展：

- `running`
- `completed`

## 5. 工具协议设计

### 5.1 `workflow_wait_for_user`

用途：

- 当模型无法继续，需要人工确认、补参、审批或补充上下文时调用

输入字段：

- `title`
- `prompt`
- `context_summary`
- `system_instruction`
- `fields`
- `client_name`
- `client_session_id`
- `expires_in_seconds`
- `metadata`

返回字段：

- `session_id`
- `status=waiting_user`
- `message`
- `ui_url`
- `fields`
- `next_action=poll`
- `poll_after_ms`
- `system_instruction`

### 5.2 `workflow_poll`

用途：

- 在进入等待态后由模型持续轮询

输入字段：

- `session_id`
- `wait_ms`

返回可能包括：

- `status=waiting_user`
- `status=submitted`
- `status=expired`
- `status=failed`

当 `status=submitted` 时，返回里会包含：

- `user_input`
- `submitted_data`

## 6. 提示词策略

### 6.1 全局系统提示词

目标是给模型建立固定行为规则：

1. 一旦缺少人工输入，就调用 `workflow_wait_for_user`
2. 调完后不能直接输出最终答案
3. 当工具返回 `waiting_user` 时继续调用 `workflow_poll`
4. 当工具返回 `submitted` 时使用用户输入继续任务
5. 当工具返回 `expired` 或 `failed` 时停止伪造流程

### 6.2 工具返回中的内置控制指令

每次工具返回时，还会给出 `system_instruction`。

它的作用是：

- 防止模型忘记继续轮询
- 防止模型在等待状态下开始总结
- 防止模型在长链路中偏离协议

这个字段虽然未必被所有客户端真正当成协议级 `system role`，但在语义上应被视为当前步骤的最高优先级控制指令。

## 7. 为什么不能只靠“同一轮一直挂起”

你希望等待 2 小时以上，这件事在桥接层可以支持，但不能把它设计成：

- 一个 HTTP 请求一直不结束
- 一个 MCP 工具调用一直不返回
- 一个客户端 agent loop 永不超时

跨客户端场景下，下面这些都不能强保证：

- 单次传输保持 2 小时不断开
- 客户端持续轮询 2 小时不被中断
- 客户端在中断后仍然自动恢复同一轮

所以正确设计应该是：

- 短轮询尽量争取“同一轮继续”
- 外部状态机保证“中断后可恢复”

## 8. 关于“只扣一次额度”

这个目标主要针对 `GitHub Copilot`。

从产品思路上，这个方案希望做到：

- 用户只发送一次 prompt
- 后续是 agent 的工具调用和后台步骤
- 人工输入通过外部网页进入
- 模型通过 `poll` 拿到输入继续执行

但要注意：

- 这不代表所有客户端都按 Copilot 的计费规则工作
- 也不代表所有客户端都能无限期维持同一个 agent 执行链

因此应该把产品目标定义成：

- 对 Copilot 优先优化“单用户 prompt 驱动的多步流程”
- 对其他客户端保证协议兼容和可恢复
