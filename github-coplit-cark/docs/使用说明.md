# 使用说明

## 1. 这个工具是干什么的

这个工具是一个 `MCP` 桥接服务，用来给大模型提供“等待人工输入后再继续任务”的能力。

它的工作方式是：

1. 模型先调用 `workflow_wait_for_user`
2. 服务创建一个等待会话
3. 服务返回一个网页地址 `ui_url`
4. 人工在网页中填写内容
5. 模型持续调用 `workflow_poll`
6. 一旦人工提交完成，模型就能拿到结构化输入并继续工作

这个工具适合的场景包括：

- 缺少部署参数，等待人工确认
- 需要审批后再继续
- 需要补充环境变量、账号、命名空间、开关项
- 需要多轮任务中人为插入确认点

## 2. 当前支持的能力

当前原型支持：

- `stdio` 方式运行的 MCP 服务
- 独立网页输入页
- 基于 `SQLite` 的等待会话持久化
- 纯文本输入
- 结构化表单输入
- 轮询式恢复

当前还不支持：

- 真实调用 GitHub Copilot Claude Opus 的任务执行器
- 鉴权和签名 URL
- 多用户隔离
- 生产级日志和审计

## 3. 启动方式

在项目根目录执行：

```bash
python -m copilot_bridge.main --web-host 127.0.0.1 --web-port 4317 --db-path bridge-runtime.db
```

参数说明：

- `--web-host`
  - Web 输入页监听地址
- `--web-port`
  - Web 输入页监听端口
- `--db-path`
  - `SQLite` 数据库文件路径
- `--sqlite-journal-mode`
  - `SQLite` journal 模式，默认使用 `PERSIST`，在某些受限文件系统或外置盘环境下比默认模式更稳定
- `--public-base-url`
  - 如果 MCP 服务和网页对外暴露的地址不是本机地址，需要显式指定这个参数
- `--default-expiry-seconds`
  - 默认等待时长，默认是 `7200` 秒，也就是 2 小时

例如，如果网页通过反向代理暴露到别的地址：

```bash
python -m copilot_bridge.main ^
  --web-host 0.0.0.0 ^
  --web-port 4317 ^
  --public-base-url https://your-host.example.com ^
  --db-path bridge-runtime.db
```

## 5. 工具列表

当前提供两个工具：

- `workflow_wait_for_user`
- `workflow_poll`

### 5.1 `workflow_wait_for_user`

作用：

- 创建一个“等待人工输入”的会话

最小调用示例：

```json
{
  "name": "workflow_wait_for_user",
  "arguments": {
    "title": "请确认部署信息",
    "prompt": "请填写生产命名空间，并确认是否开启 canary。"
  }
}
```

结构化字段示例：

```json
{
  "name": "workflow_wait_for_user",
  "arguments": {
    "title": "确认部署参数",
    "prompt": "请补充部署所需字段。",
    "context_summary": "当前只缺少生产环境参数。",
    "fields": [
      {
        "name": "namespace",
        "label": "生产命名空间",
        "type": "text",
        "required": true,
        "placeholder": "prod-payments"
      },
      {
        "name": "canary",
        "label": "是否开启 canary",
        "type": "boolean",
        "required": true
      },
      {
        "name": "strategy",
        "label": "发布策略",
        "type": "select",
        "required": true,
        "options": [
          "blue",
          "green"
        ]
      }
    ],
    "client_name": "github-copilot",
    "expires_in_seconds": 7200
  }
}
```

返回示例：

```json
{
  "session_id": "sess_xxx",
  "status": "waiting_user",
  "message": "Human input is required before the workflow can continue.",
  "ui_url": "http://127.0.0.1:4317/s/abc123",
  "fields": [
    {
      "name": "namespace",
      "label": "生产命名空间",
      "type": "text",
      "required": true
    }
  ],
  "next_action": "poll",
  "poll_after_ms": 3000,
  "system_instruction": "The workflow is waiting for human input. Do not finalize the task. Call workflow_poll again after the suggested delay."
}
```

### 5.2 `workflow_poll`

作用：

- 轮询前面创建的等待会话

调用示例：

```json
{
  "name": "workflow_poll",
  "arguments": {
    "session_id": "sess_xxx",
    "wait_ms": 3000
  }
}
```

可能返回 4 类状态：

- `waiting_user`
- `submitted`
- `expired`
- `failed`

当返回 `waiting_user` 时，说明用户还没提交：

```json
{
  "session_id": "sess_xxx",
  "status": "waiting_user",
  "message": "Still waiting for human input.",
  "fields": [
    {
      "name": "namespace",
      "label": "生产命名空间",
      "type": "text",
      "required": true
    }
  ],
  "next_action": "poll",
  "poll_after_ms": 3000,
  "system_instruction": "The workflow is waiting for human input. Do not finalize the task. Call workflow_poll again after the suggested delay."
}
```

当返回 `submitted` 时，说明用户已经填写完成：

```json
{
  "session_id": "sess_xxx",
  "status": "submitted",
  "user_input": "生产命名空间: prod-payments\n是否开启 canary: True\n发布策略: blue",
  "submitted_data": {
    "namespace": "prod-payments",
    "canary": true,
    "strategy": "blue"
  },
  "message": "Human input is available.",
  "next_action": "continue_task",
  "system_instruction": "Human input is now available. Use the returned input to continue the task. Do not ask the user to repeat it unless validation fails."
}
```

## 6. 用户网页输入流程

当 `workflow_wait_for_user` 返回 `ui_url` 后：

1. 打开这个网页
2. 页面会显示：
   - 标题
   - 提示内容
   - 上下文摘要
   - 输入框或结构化字段
3. 用户填写并提交
4. 提交后，服务会把内容写回会话状态
5. 模型下一次调用 `workflow_poll` 时就能拿到结果

### 6.1 token 是什么，丢了怎么找

这个项目没有单独的“登录 token”概念。网页访问用的 token 是每次等待会话创建时随机生成的，并且已经包含在返回的 `ui_url` 里，例如：

```text
http://127.0.0.1:4317/s/abc123
```

这里最后一段 `abc123` 就是这次会话的 token。

如果你丢了这个地址，可以直接列出数据库里最近的会话：

```bash
python -m copilot_bridge.main --db-path bridge-runtime.db --list-sessions
```

它会打印最近会话的：

- `session_id`
- `status`
- `ui_url`
- `token`

如果你之前使用的是旧的 `bridge.db`，并且崩溃后留下了 `bridge.db-journal`，建议改用新的 `db-path`，例如 `bridge-runtime.db`，避免旧残留文件继续影响写入。

## 7. 推荐的提示词用法

为了让模型稳定使用这个工具，建议在系统提示词里加入类似规则：

```text
当任务缺少人工输入、审批或确认时，必须调用 workflow_wait_for_user。
调用后不得直接输出最终答案。
如果工具返回 waiting_user，继续调用 workflow_poll。
如果工具返回 submitted，使用返回的 submitted_data 继续任务。
如果工具返回 expired 或 failed，不要伪造结果。
如果工具返回 system_instruction，应将其视为当前工作流步骤的最高优先级控制指令。
```

## 8. 一个完整的最小流程

### 第一步：模型调用等待工具

```json
{
  "name": "workflow_wait_for_user",
  "arguments": {
    "title": "请确认部署参数",
    "prompt": "请填写生产命名空间。",
    "fields": [
      {
        "name": "namespace",
        "label": "生产命名空间",
        "type": "text",
        "required": true
      }
    ]
  }
}
```

### 第二步：工具返回网页地址

模型拿到：

- `session_id`
- `ui_url`
- `status=waiting_user`

### 第三步：用户打开网页并提交

用户在网页中填写：

```text
prod-payments
```

### 第四步：模型轮询

```json
{
  "name": "workflow_poll",
  "arguments": {
    "session_id": "sess_xxx",
    "wait_ms": 3000
  }
}
```

### 第五步：工具返回结构化结果

```json
{
  "session_id": "sess_xxx",
  "status": "submitted",
  "submitted_data": {
    "namespace": "prod-payments"
  }
}
```

### 第六步：模型继续真正任务

此时模型可以继续后续逻辑，例如：

- 生成部署命令
- 补全配置
- 继续调用其他工具

## 9. 当前边界

要注意这几个现实限制：

### 9.1 不能强保证“同一轮永远不结束”

虽然桥接层支持 2 小时以上等待，但不能跨所有客户端强保证：

- 一个 MCP 请求一直保持打开
- 一个 agent loop 永远不超时
- 一个客户端自动恢复到原来的同一轮状态

所以更现实的目标是：

- 尽量在同一轮里轮询继续
- 即使中断，也能靠 `session_id` 恢复

### 9.2 桥接层会话独立于客户端内部会话

这个工具不会复用：

- VS Code 内部聊天历史
- Codex CLI 内部会话持久化
- Claude Code CLI 内部状态
- OpenCode 内部状态

它只会记录自己的等待会话，并通过：

- `session_id`
- `client_name`
- `client_session_id`

去做关联。

### 9.3 还没有接真实上游模型执行器

当前原型只实现了：

- 等待人工输入
- 表单网页
- 状态恢复

还没有把“真实任务执行”和“调用 GitHub Copilot Claude Opus”接进来。
